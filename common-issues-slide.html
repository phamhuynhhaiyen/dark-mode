Common Implementation Issues

Where dark mode usually breaks down in real production apps

◐

Flash of Incorrect Theme

Why it happens

The app renders in light mode first, then JavaScript switches to dark mode after the page is already visible.

What to do

Apply the initial theme before first paint and hydrate from saved preference or system preference as early as possible.

#

Hardcoded Colors in Components

Why it happens

Components still contain values like #fff or #111827 instead of using semantic tokens.

What to do

Make components consume semantic tokens only, then keep theme-specific values inside the theme layer.

⊡

Missing Token Coverage

Why it happens

The system covers background and text, but forgets hover, focus, disabled, border, shadow, or chart colors.

What to do

Audit UI states and build token coverage for interaction states, elevation, data visualization, and semantic feedback colors.

⇄

Third-Party Components Drift

Why it happens

Charts, editors, embedded widgets, and UI libraries do not automatically follow your token system.

What to do

Create adapter layers or theme bridges so external components receive the same semantic theme values.

!

Implementation Checklist

Most dark mode bugs come from incomplete coverage, not from the toggle itself.

Initial theme

Set the right theme before the first visible render.

Token discipline

Keep hardcoded colors out of components.

State coverage

Include hover, focus, disabled, border, and shadow tokens.

Third-party audit

Verify charts, editors, and external widgets follow the theme.

theme = "dark"

Design System Architecture  |  How the theme system is built
▣

Components

Components consume semantic tokens only, not hardcoded theme values.

.card {
  background: var(--surface-primary);
  color: var(--text-primary);
  border: 1px solid var(--border-subtle);
}
🏷

Semantic Tokens

Meaningful tokens that describe UI roles such as surface, text, border, and accent.

--surface-primary
--text-primary
--border-subtle
--surface-secondary
--text-secondary
--accent-primary
◫

Theme Layer

Each theme maps semantic tokens to primitive design values.

Light Theme
--surface-primary → var(--color-white) --text-primary → var(--color-gray-900)
Dark Theme
--surface-primary → var(--color-gray-900) --text-primary → var(--color-white)
High Contrast
--surface-primary → var(--color-black) --text-primary → var(--color-white)
⬚

Primitive Tokens

Raw design values such as colors, spacing, radius, and shadows.

--color-white
#FFFFFF
--color-gray-900
#121212
--color-gray-100
#F5F5F5
--color-blue-500
#3B82F6
--color-green-500
#22C55E
Why This Works
+
Scalable

Add new themes without changing component code.

✓
Consistent

All components follow the same token system.

↺
Maintainable

Update one token mapping and the UI updates everywhere.

⚡
Efficient

Theme switching usually updates styles without changing component structure.

💡

The Key Idea

Switching theme means changing token mappings, not changing components.
